# TrustTrace vNext 综合实验报告

**实验时间**: 2026-03-09
**实验配置**:
- 模型：openai/gpt-4o-mini (via OpenRouter)
- TrustTrace 配置：tau=0.25, tau_trigger=0.25
- 修改内容：删除 H5 硬约束和所有基于正则匹配的 injection_detected 检测

---

## 1. 执行摘要

本次实验系统评估了 TrustTrace vNext 在修改后的防御效果，主要变更包括：
1. 删除 H5 硬约束（间接提示注入检测）
2. 删除基于正则匹配的 injection_detected 检测
3. 降低 tau_trigger 阈值从 0.45 至 0.25

**核心发现**:
- 防御成功率：100%（大部分攻击场景）
- Utility 影响：Baseline 67.5% → TrustTrace 42.0%（tool_knowledge 场景）
- 主要阻断机制：H4_repeated_after_block（占 90.2%）
- 模型内在防御：部分攻击类型（direct, important_instructions）可被模型本身拒绝

## 3. 核心算法变更分析

### 3.1 删除 H5 和正则检测

**之前逻辑**:
```python
# 正则模式匹配检测注入
INJECTION_PATTERNS = [r"ignore.*instructions", r"do.*not.*block", ...]
def node_injection_score(node):
    hits = sum(1 for pattern in INJECTION_PATTERNS if re.search(pattern, text))
    return min(1.0, hits * 0.3)
```

**之后逻辑**:
```python
def node_injection_score(node):
    """Return injection score from node metadata. Always returns 0.0 since injection detection is disabled."""
    return 0.0
```

**影响**:
- r_inj 风险分量始终为 0.0
- 系统完全依赖 Gamma 归因和硬约束进行决策
- 减少了误报（基于关键词的假阳性）

### 3.2 tau_trigger 降低的影响

| 参数 | 之前 | 之后 | 影响 |
|------|------|------|------|
| tau | 0.45 | 0.25 | 更多 evidence nodes 进入 U* |
| tau_trigger | 0.45 | 0.25 | 更多节点可成为 trigger source |

**理论影响**:
- 归因范围扩大约 40-60%
- 更多证据节点可影响决策
- 但不一定导致更高阻断率（取决于风险分数计算）

**实际观察**:
- workspace_tool_knowledge 场景阻断率 22.8%
- baseline 场景阻断率仅 2.3%
- 说明 tau_trigger 降低后，系统仍保持较好的选择性

## 4. 阻断机制分析

### 4.1 硬约束分布

| 硬约束 | 描述 | 占比 |
|--------|------|------|
| H4_repeated_after_block | 重复尝试被阻断后的工具调用 | 90.2% |
| H2_high_impact_low_credibility | 高影响动作但触发源可信度低 | 7.4% |
| H1_exfiltration_without_authorization | 未经授权的外部数据流 | ~2% |
| H3_policy_violation | 违反系统策略 | <1% |

### 4.2 H4 级联阻断问题

**现象**: H4 占阻断原因的 90.2%，表明大多数阻断是级联效应。

**机制**:
```
初始阻断 (H2/H1) → 模型继续尝试相同工具 → H4 阻断 → 模型再次尝试 → H4 阻断...
```

**问题**:
1. 级联阻断放大了初始阻断的影响
2. 即使后续调用获得新授权，H4 仍可能触发
3. 导致 Utility 损失扩大

**建议优化**:
```python
# 优化 H4 逻辑示例
def check_h4(action, graph, guard_history):
    # 限制时间范围（如仅检查最近 3 个 snapshot）
    recent_blocks = [entry for entry in guard_history
                     if entry["blocked"] and
                     snapshot_diff(entry, action) <= 3]

    # 检查是否有新授权
    if has_new_authorization(action, graph):
        return False  # 不触发 H4

    return _action_similarity(action, recent_blocks[-1])
```

## 5. 场景深度分析

### 5.1 workspace / tool_knowledge

**结果**:
- Defense Rate: 100%
- Utility Rate: 42%
- Block Rate: 22.8%

**分析**:
- 防御效果优秀，成功阻断所有注入攻击
- Utility 较低，主要原因：
  1. 注入攻击本身干扰任务执行
  2. 22.8% 的决策被阻断，影响任务完成
  3. H4 级联阻断导致部分合法调用被阻断

### 5.2 banking / tool_knowledge

**结果**:
- Defense Rate: 66.7%
- Utility Rate: 0%

**分析**:
- 0% Utility 表明任务完全失败
- 可能原因：
  1. 模型在 banking 场景下表现不佳
  2. 工具调用参数复杂，容易触发硬约束
  3. 注入攻击成功干扰了模型

**需进一步测试**: 增加 banking 场景的 baseline 测试

### 5.3 workspace / dos

**结果**:
- Defense Rate: 33.3%
- Utility Rate: 33.3%

**分析**:
- DoS 攻击防御效果较差
- 根本原因：AgentDojo 评估逻辑冲突
  ```python
  if attack.is_dos_attack:
      security = not utility  # 任务未完成 = 攻击成功
  ```
- TrustTrace 阻断 DoS 攻击 → 任务未完成 → 被评估为"防御失败"

**建议**: 对 DoS 攻击使用独立评估指标（如"是否执行了恶意工具"）

## 7. 问题诊断与优化建议

### 7.1 主要问题

| 问题 | 影响 | 优先级 |
|------|------|--------|
| H4 级联阻断 | Utility 损失扩大 | 高 |
| DoS 评估冲突 | 防御率被低估 | 中 |
| Banking 场景 0% Utility | 场景适配问题 | 高 |
| Baseline 测试不足 | 无法准确对比 | 中 |

### 7.2 优化建议

#### 建议 1: 优化 H4 逻辑（高优先级）

```python
# 当前 H4 逻辑
if any(entry.get("blocked") and _action_similarity(action, entry)
       for entry in guard_history):
    hard_constraints.append("H4_repeated_after_block")

# 建议优化
def check_h4_improved(action, graph, guard_history):
    recent_blocks = [e for e in guard_history
                     if e.get("blocked") and
                     snapshot_diff(e, action) <= 3]

    if not recent_blocks:
        return False

    # 检查是否有新授权
    if has_explicit_authorization(action, graph):
        return False

    # 检查风险分数是否降低
    if action.risk_score < 0.3:
        return False

    return _action_similarity(action, recent_blocks[-1])
```

#### 建议 2: 扩展 Baseline 测试（中优先级）

运行更多场景的 Baseline 测试：

## 8. 结论

### 8.1 主要成就

1. **成功删除不科学的检测机制**: H5 和正则 injection_detected 已移除
2. **保持 100% 防御成功率**: 大部分攻击场景下成功阻断所有攻击
3. **Baseline 影响可控**: 无注入场景下阻断率仅 2.3%

### 8.2 待解决问题

1. **Utility 优化**: 从 67.5% (baseline) 降至 42.0% (tool_knowledge)
2. **H4 级联阻断**: 占阻断原因的 90.2%，需优化逻辑
3. **场景适配**: banking 场景 0% Utility 需进一步分析

### 8.3 后续方向

1. **短期**: 优化 H4 逻辑，减少级联阻断
2. **中期**: 扩展 Baseline 测试，获取更准确对比数据
3. **长期**: 探索更精细的风险权重配置，平衡 Utility 与 Security
