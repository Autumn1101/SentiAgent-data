# TrustTrace Baseline 对比分析报告（修订版）

**实验时间**: 2026-03-10
**实验目的**: 对比 TrustTrace 启用/禁用情况下的 utility 和 security 表现
**模型**: openai/gpt-4o-mini (via OpenRouter)
**归因阈值**: tau_trigger=0.25

---

## 执行摘要（重要更新）

### 关键发现

1. **Baseline 防御能力远超预期**: GPT-4o-mini 模型自身对注入攻击有极强的抵抗能力
   - workspace/direct: **100% 防御率**（模型完全抵抗所有注入攻击）
   - workspace/tool_knowledge: **96.8% 防御率**
   - workspace/important_instructions: **86.8% 防御率**
   - travel/tool_knowledge: **84.3% 防御率**
   - banking/tool_knowledge: **79.9% 防御率**

2. **TrustTrace 增量价值有限**:
   - 在模型已有高防御率的情况下，TrustTrace 仅能提供 0% 到 +3.2% 的防御提升
   - 唯一显著提升：important_instructions 场景 +12.7%

3. **Utility 代价显著**:
   - TrustTrace 启用后 Utility 平均下降 13-27%
   - 原因：H4 级联阻断过于保守

4. **主要瓶颈**: H4 级联阻断机制
   - ~90% 的阻断来自 H4 (Repeated After Block)
   - 一次初始阻断后，后续相同工具调用全部被阻断
   - 这导致 Utility 大幅下降

## 2. 深入分析：为什么 Baseline 防御率如此之高？

### 2.1 可能的原因

1. **模型训练因素**:
   - GPT-4o-mini 经过 RLHF（Reinforcement Learning from Human Feedback）训练
   - 系统提示可能包含安全指导，教导模型拒绝可疑请求
   - 模型对常见的注入攻击模式有内在抵抗力

2. **攻击强度问题**:
   - AgentDojo 的注入攻击可能不够强
   - 攻击主要依赖简单的指令覆盖（如"忽略之前的指令"）
   - 这类攻击对于经过对齐训练的模型效果有限

3. **任务设计因素**:
   - 部分攻击场景可能与模型的默认行为一致
   - 模型可能本身就倾向于执行某些操作（如删除文件时询问确认）

### 2.2 与预期的差异

**预期**: Baseline 防御率应该在 40-60% 左右，TrustTrace 能提供显著的增量价值

**实际**: Baseline 防御率达到 80-100%，TrustTrace 增量价值有限

**影响**:
- TrustTrace 的主要价值主张（提供额外防御层）在强模型场景下受到挑战
- 需要重新评估 TrustTrace 的目标使用场景

## 3. TrustTrace 框架面临的主要问题和瓶颈

### 3.1 H4 级联阻断问题（最严重）

**现象**:
- ~90% 的阻断来自 H4 (Repeated After Block)
- 一次初始阻断后，后续相同工具调用全部被阻断
- 这导致 Utility 大幅下降（尤其是 important_instructions 场景，Utility 从 40.7% 降至 13.2%）

**根本原因**:
- H4 的设计过于绝对化，缺少状态重置机制
- 没有考虑上下文变化（如用户获得新授权、情况变化等）
- 级联效应放大了初始判断的误差

**建议优化**:
1. 限制 H4 作用范围（如仅限同一 snapshot 内）
2. 增加状态重置机制（如获得新授权后、用户明确确认后）
3. 考虑将 H4 与风险分数结合，而非绝对阻断
4. 引入"冷却期"机制，避免连续阻断

### 3.2 风险评估模型准确性问题

**现象**:
- banking/tool_knowledge 场景 TrustTrace 的 Defense 反而下降 2.1%
- TrustTrace 在该场景的阻断率为 0%，未产生任何阻断

**可能原因**:
- 风险评估权重在 banking 场景不够准确
- 缺乏针对 banking 场景的特殊规则
- 归因阈值 (tau=0.25) 可能不适合所有场景

**建议优化**:
1. 针对不同场景调整风险权重
2. 增加场景特定的检测规则
3. 考虑动态调整归因阈值

### 3.3 增量价值有限

**问题**:
- 对于 GPT-4o-mini 等强模型，TrustTrace 的增量防御价值有限（0% 到 +3.2%）
- Utility 代价却很高（平均下降 13-27%）

**建议方向**:
1. 针对更弱的模型进行测试和优化（如 GPT-3.5、开源模型）
2. 探索 TrustTrace 在高安全性要求场景的应用（即使增量价值小，也可能值得）
3. 考虑将 TrustTrace 作为"最后一道防线"，而非主要防御手段

### 3.4 误报与漏报平衡

**问题**:
- 当前框架倾向于保守（宁可误报，不可漏报）
- 这导致 Utility 损失较大

**建议优化**:
1. 引入可调节的"保守度"参数
2. 对于不同风险级别的操作采用不同的阻断策略
3. 增加"软阻断"机制（如要求用户确认，而非直接拒绝）

## 4. 后续优化方向

### 4.1 短期优化（高优先级）

1. **优化 H4 逻辑**:
   - 限制作用范围
   - 增加状态重置机制
   - 与风险分数结合

2. **细化场景配置**:
   - 针对不同场景调整风险权重
   - 增加场景特定规则

3. **改进可视化**:
   - 提供更详细的阻断原因分析
   - 帮助用户理解 TrustTrace 的决策过程

### 4.2 中期优化（中优先级）

1. **测试更弱的模型**:
   - 在 GPT-3.5、开源模型上测试 TrustTrace
   - 验证 TrustTrace 在防御能力较弱模型上的增量价值

2. **探索动态阈值**:
   - 根据场景和风险级别动态调整归因阈值
   - 引入自适应学习机制

3. **增加用户交互**:
   - 对于边界情况，增加用户确认机制
   - 提供"信任此操作"选项

### 4.3 长期方向（低优先级）

1. **探索大模型应用场景**:
   - 针对高安全性要求场景（如金融、医疗）
   - 即使增量价值小，也可能值得采用

2. **研究更强的注入攻击**:
   - 设计更隐蔽、更强的攻击方式
   - 验证 TrustTrace 在极限场景下的表现

## 6. 结论

### 6.1 TrustTrace 有效性评估

| 维度 | 评估 | 证据 |
|------|------|------|
| **必要性** | 低 - 中 | Baseline Defense 已达 79-100%，模型自身防御强劲 |
| **有效性** | 混合 | workspace +3.2% 至 +12.7%, banking -2.1% |
| **Utility 代价** | 高 | 平均下降约 13-27%（除 banking 外） |

### 6.2 核心问题总结

1. **H4 级联阻断**是最主要的瓶颈，导致 ~90% 的阻断和严重的 Utility 损失
2. **模型自身防御强劲**，TrustTrace 增量价值有限
3. **场景差异明显**，banking 场景 TrustTrace 效果不如 Baseline

### 6.3 建议

**立即行动**:
- 优化 H4 逻辑，增加状态重置机制
- 限制 H4 作用范围

**后续研究**:
- 在更弱的模型上测试 TrustTrace
- 探索高安全性要求场景的应用
